29 /**
30  * \mainpage
31  *
32  * \section intro_sec Introduction
33  *
34  * The CVODE Trajectory Generator is a command-line program based on
35  * the CVODE library which
36  * integrates a set of differential equations to calculate a trajectory
37  * in phase space. The initial conditions for the system are given on
38  * the command line and the differential equations are provided by a
39  * "problem" object. The supported problem types are listed at the end
40  * of this section.
41  *
42  * The Generator is capable of integrating the differential equations
43  * for only one set of initial conditions per execution. The initial
44  * conditions are specified on the command-line, either by the
45  * \c --initial-vector option or by options specific to a problem. The
46  * format of output data may vary with the problem being used, but all
47  * current problems output columns of data (separated by tabs)
48  * representing the state of
49  * the system at many intermediate points during the integration.
50  *
51  * \section example_sec The Escape problem: an example
52  *
53  * The Generator was originally written for the Escape problem. In this
54  * problem, we simulate the path taken by a small particle of infinitesimal
55  * mass under the Newtonian gravitational influence of two massive
56  * bodies which orbit each other in concentric circles. The problem is
57  * a restricted version of the classical three-body problem; it is
58  * implemented in the RestrictedThree class.
59  *
60  * In the current build system, \c escape and \c rthree are identical
61  * executables which integrate the RestrictedThree problem. With the
62  * default parameters and initial conditions, \c escape produces output
63  * like this:
64  *
65 \verbatim
66  $ ./escape
67 #t x y
68 0 0.8 0
69 0.0321 0.867814 0.0224674
70 0.0641 0.923033 0.0448505
71 0.0961 0.97147 0.0672085
72 0.1281 1.01532 0.0895327
73 0.1602 1.05587 0.111886
74 0.1923 1.09369 0.134191
75 0.2244 1.12926 0.156445
76 0.2565 1.16294 0.178643
77 ...
78 \endverbatim
79  *
80  * The first column is the system time, where \f$t = 0\f$ is the start
81  * of integration, and \f$t = 2\pi\f$ marks the time when the 2-body system
82  * completes its first revolution. The next two columns give the familiar
83  * Cartesian coordinates of the projectile's position in the plane. When
84  * additional options are specified, additional columns may appear:
85  *
86 \verbatim
87  $ ./escape -E -G -X -0.24 -Y -0.584
88 #t x y e
89 0 0.661989 -0.092494 0
90 0.0321 0.638866 -0.1253 8.50784e-11
91 0.0641 0.619952 -0.145911 1.07883e-10
92 0.0961 0.603042 -0.159546 1.13102e-10
93 0.1281 0.587256 -0.16827 1.22007e-10
94 0.1602 0.572063 -0.173198 1.23826e-10
95 0.1923 0.557199 -0.174981 1.24331e-10
96 0.2244 0.542441 -0.174081 1.24602e-10
97 0.2565 0.527617 -0.170828 1.2514e-10
98 ...
99 \endverbatim
100  *
101  * Here the initial conditions are specified by the \c -X and \c -Y
102  * options. The \c -G option specifies how the \c -X and \c -Y options
103  * are to be interpreted, and the \c -E option adds an additional column
104  * giving the deviation from initial pseudoenergy. (all these options are
105  * documented in RestrictedThree::getOptionString())
106  *
107  * The frequency of output is controlled with the \c -N option and the
108  * upper bound of integration is specified with \c -e. For example:
109 \verbatim
110  $ ./escape -e 44 -N 4
111 #t x y
112 0 0.8 0
113 11.0001 2.28949 3.84313
114 22.0002 -0.592399 0.147827
115 33.0002 -4.77448 0.0824081
116 44 -3.82037 -1.39882
117  $
118 \endverbatim
119  *
120  * The \c escape executable and others like it are infrequently used
121  * alone like this. Usually they are used in the so-called "dispatch"
122  * scripts developed for the Escape problem, which can be used to run
123  * \c escape millions of times with different
124  * intial conditions and distribute the computational task across
125  * many computers using Apple's Xgrid software.
126  *
127  * \section param_sec Command-line parameters
128  *
129  * The generator supports many command-line parameters. The following
130  * parameters are valid for all problems:
131  *
132  * - \c -p: \n
133  * Select problem to simulate. This is currently not supported; for now
134  * each problem produces its own executable.
135  *
136  * - \c -I, \c --integration-method: \n
137  * Select integrator. This should be either "bdf" or "adams".
138  * - CVODE Manual: <a href="http://www.llnl.gov/casc/sundials/documentation/cv_guide/node6.html#SECTION00651000000000000000">5.5.1 CVODE initialization and deallocation functions</a> - CVodeCreate()
139  *
140  * - \c -i, \c --iterator:\n
141  * Select iterator. This should be either "newton" or "functional".
142  * - CVODE Manual: <a href="http://www.llnl.gov/casc/sundials/documentation/cv_guide/node6.html#SECTION00651000000000000000">5.5.1 CVODE initialization and deallocation functions</a> - CVodeCreate()
143  *
144  * - \c -v, \c --initial-vector: \n
145  * A vector of initial conditions, separated by commas. The
146  * list must be of the length required by the problem in use.
147  *
148  * - \c -r, \c --relative-tolerance: \n
149  * Relative tolerance. Values around \c 1e-8 or \c 1e-12 have
150  * been found to work well.
151  * - CVODE Manual: <a href="http://www.llnl.gov/casc/sundials/documentation/cv_guide/node6.html#SECTION00652000000000000000">5.5.2 Advice on choice and use of tolerances</a>
152  *
153  * - \c -T:\n
154  * Absolute tolerances, as a comma-separated list. The list
155  * must be of the length required by the problem in use.
156  * - CVODE Manual: <a href="http://www.llnl.gov/casc/sundials/documentation/cv_guide/node6.html#SECTION00652000000000000000">5.5.2 Advice on choice and use of tolerances</a>
157  *
158  * - \c -t, \c --absolute-tolerance: \n
159  * Absolute tolerance, as a scalar. The list of absolute tolerances
160  * is filled with copies of this value.
161  * - CVODE Manual: <a href="http://www.llnl.gov/casc/sundials/documentation/cv_guide/node6.html#SECTION00652000000000000000">5.5.2 Advice on choice and use of tolerances</a>
162  *
163  * - \c -b, \c --begin: \n
164  * Beginning time. The unit of time is defined by the
165  * problem in use. It is generally safe to leave this at its default;
166  * in fact, the generator may have trouble if this parameter is used.
167  * Default: 0
168  *
169  * - \c -e, \c --end: \n
170  * Ending time.
171  *
172  * - \c -s, \c --timestep: \n
173  * Internal timestep for CVODE integration. Smaller values allow the
174  * integrator to go farther without dying with a "too much work" error.
175  * But make this too small, and you gain no precision but lose
176  * performance.
177  *
178  * - \c -N, \c --number-points, \c --data-length: \n
179  * Number of points to record. Default: 1000
180  *
181  * - \c -L, \c --logtime-start: \n
182  * Enable logarithmic time for data recording and set \f$t_0\f$.
183  *
184  * - \c -n, \c --logtime-n: \n
185  * Enable logarithmic time for data recording and set \f$n\f$ so
186  * \f$r=2^{1/n}\f$
187  *
188  * - \c -m, \c --max-internal-steps: \n
189  * Maximum number of internal CVODE steps to take per timestep. Bigger
190  * values for this parameter tend to eliminate "too much work" errors
191  * and seem to have no negative effect on performance. Default: 1000000
192  *
193  * Other options are specified by the particular problem in use.
194  *
195  * - RestrictedThree::getOptionString()
196  * - SingleBody::getOptionString()
197  * - TwoBody::getOptionString()
198  * - CurvedTwoBody::getOptionString()
199  */

353 /**
354  * Split a string into a vector of \c realtype values.
355  *
356  * In the following example, the vector \a v is filled with the four
357  * values 1.0, 2.7, 3.2, and 4.03:
358  *
359 \code
360 vector<realtype> v;
361 v = split_realtype(",", "1.0,2.7,3.2,4.03");
362 \endcode
363  *
364  * The implementation does not try to be particularly efficient. It
365  * does, however, do a pretty good job of being correct.
366  *
367  * @param sep Delimeter on which to split. \a sep is stripped
368  * from the output.
369  * @param text Text to split.
370  */
371 vector<realtype> split_realtype(const string &sep,string text)
372 {
373  vector<realtype> numbers;
374  string::size_type end;
375  do
376  {
377  end = text.find(sep);
378  if (end == string::npos)
379  end = text.length() + 1;
380 
381  numbers.push_back(atof(text.substr(0,end).c_str()));
382  text.replace(0,end+sep.length(),"");
383 
384  } while (text.length());
385  return numbers;
386 }
